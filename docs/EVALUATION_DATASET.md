# Evaluation Dataset

This document describes the planned evaluation-set structure for the Nuosu Bburma OCR project.

Final dataset links, scale, category counts, and evaluation results will be added after review and rerun.

## Purpose

The evaluation set is designed to test whether a model can handle practical Nuosu Bburma OCR scenarios rather than only memorize one book or one synthetic style.

It should cover:

- pure Nuosu Bburma text;
- Yi-Han mixed text;
- old printed material;
- line-level OCR;
- selected page or region OCR;
- limited handwriting cases;
- punctuation, digits, and symbols that appear in real materials.

## Required Dataset Package

The final evaluation package should include:

```text
images/
annotations.jsonl
dataset_summary.json
task_description.md
evaluate.py
README.md
```

If the full dataset is hosted externally, this repository should still provide:

- dataset description;
- sample rows;
- annotation format;
- evaluation script or script link;
- access instructions.

## Annotation Format

Recommended JSONL fields:

```json
{
  "id": "sample_id",
  "image": "relative/path/to/image.png",
  "text": "ground truth text",
  "split": "eval",
  "source_type": "real|synthetic|handwriting|old_print|mixed",
  "layout_type": "line|region|page",
  "notes": ""
}
```

The final schema can be adjusted after the rerun, but source type and layout type should remain explicit.

## Metrics

Primary metric:

- NED, normalized edit distance.

Additional diagnostic checks:

- Yi-only behavior;
- Han-only or mixed-script behavior;
- punctuation and digit drift;
- Latin/ASCII drift;
- LaTeX-like or formula-style hallucination;
- long-prediction failures;
- empty or replacement-style outputs.

## Review Rules

- Evaluation samples should not be used as training targets.
- Real and synthetic samples should be traceable.
- Synthetic ratio must be stated conservatively.
- Handwriting should be reported as a difficult sub-scenario.
- Line-level and page/region-level performance should not be merged without explanation.

## Rerun Policy

The final report should include only metrics generated by the final rerun.

Until then:

- result tables remain `TBD`;
- old local metrics may be used only as internal references;
- final model selection must be checked against the rerun output.
