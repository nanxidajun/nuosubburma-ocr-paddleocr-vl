训练数据全部为独立程序合成；1,030 张真实评估图片及其人工 GT 只用于最终评估，不进入训练或开发数据。

## 训练数据要教会模型什么

彝文缺乏可直接用于 OCR 训练的自然文本数据，因此训练数据由字符表与版式规则**程序合成**——模型学习的是字符与版式规律，而不是记忆具体句子。围绕项目目标（整页端到端、混排识别、多场景），合成数据分两层设计：

**教基础**

- **认识所有字符**：1,165 个规范彝文字符全部覆盖学习；3,500 常用汉字集与常用字母、数字、符号适度学习；以常见彝文字体和汉字字体为渲染基础。
- **认识页面结构**：学习诗歌、文章、字典等常见版式，从单行、区域到整页，合成多行、多栏和交错混排，让版块坐标遵循自然阅读顺序。

**教难点**

- **抗干扰能力**：在清晰字形上叠加通用印刷与扫描退化（模糊、降采样、噪声、墨色变化、轻度残缺），并让模型学会不把噪点识别为字符。
- **形近字分辨能力**：第一阶段已把完整形近字表注入普通 OCR 上下文，建立通用辨识基础；第二阶段再加入严格配对的 A/B 页面与 byte-token 级辅助约束，继续细化局部笔画差异（见 §6）。

下面各节说明这两层如何落地、如何标注、如何质控与审计。

## 1. 数据来源与边界

合成数据用于补足真实材料难以稳定覆盖的字符、字体、版式和视觉退化，**不用于证明模型在真实数据上的能力**——那由独立的真实评估集衡量。允许进入生成器的输入只有规范字符表、锁定字体、生成器代码和预先冻结的通用形近字表。

| 可使用 | 禁止使用 |
|---|---|
| 规范彝文 Unicode 字表（1,165 音节） | 评估图片及其裁切图 |
| 常用汉字表 | 评估 GT |
| 锁定字体文件 | 模型在评估集上的预测 |
| 训练侧文本与版式规则 | 评估错误、错误统计、页级结果 |
| 独立生成的通用形近字表 | 按评估错例反向选择字符、字体或退化条件 |

构建不读取评估书名、样本 ID 或来源目录。开发数据同样由独立随机种子合成，并在训练前冻结。

## 2. 构建流程与可复现性

> **生成脚本已随仓库开源**，位于 [`training_data/`](../training_data/)（含按 SHA-256 锁定的核心生成器与依赖模块，见下表）。字符表与锁定字体为外部输入、不随仓库发布（字体受商用许可约束），字体各自遵循其许可。

构建链路固定，每一步的输入与产物都被记录：

```text
规范字符与符号来源
  → 确定性文本与版式采样
  → 锁定字体渲染
  → 受控视觉退化
  → 图像 / 标签配对
  → 清单、哈希与污染审计
  → 训练集或合成开发集
```

每次正式构建使用独立 build ID，记录生成器版本、输入来源、随机种子、类别配置、输出行数和 SHA-256。审计通过后即作为**不可变构建**使用，训练期间不临时改图或改标签。每条样本保留以下 provenance 字段：

```text
sample_id · text_source / character_source · font_id
layout_family · random_seed · degradation_profile
operation_order · image_sha256
```

随仓库开源的生成脚本及其 SHA-256（另含 `build_shape_pairs_v3.py`、`build_shape_pairs.py`、`confusable_injection.py` 三个依赖模块）：

| 脚本 | SHA-256 | 说明 |
|---|---|---|
| `build_synth_layout_post_v3.py` | `aef03da9…4517a2` | 第一阶段主合成器 |
| `build_post_v3_addons.py` | `941665ff…7ec2fd38` | 第一阶段附加通道（Latin / 残墨） |
| `assemble_post_v3_dataset.py` | `7d664590…0fd5b83d` | 组装训练包 |
| `post_v3_degradation_policy.py` | `fe40fd06…85238930` | 视觉退化策略 |
| `build_shape_context_contrast_v1.py` | `c25ee02b…e143dc7d2` | 第二阶段 A/B 版面生成逻辑（**公开版为设计校验版，单次上限 64 组**） |

> **关于可复现的边界（如实说明）：** 上表脚本公开的是完整生成逻辑，可逐样本复现其产物、哈希与版面规则。其中第二阶段的 `build_shape_context_contrast_v1.py` 公开的是**设计校验版**，为防误用单次限 64 组；生成正式 2,162 组 / 4,324 张的构建，以及配套的 `build_manifest.json` / `training_gate.json` / `FORMAL_BUILD_GATE.json` 审计门，运行在单独锁定的正式授权下，其完整审计产物不随公开仓库发布。因此公开材料足以复核**方法与规则**，但不足以一键重跑出正式全量数据集。

每样本从一个总 seed 派生文本、版式、渲染和退化子序列，因此多线程渲染顺序变化不会改变样本，相同 seed 的重复构建可逐样本复现，并产生一致的清单和图像哈希。

## 3. 标注规范（annotation guideline）

合成图片的标签直接来自生成器渲染时使用的文本，无需二次人工转写；但仍执行明确的标注规则：

1. 标签与实际渲染字符**逐字符一致**，不做翻译、纠错、同义替换或 Unicode 兼容字符替换；
2. 标题、正文、编号、Latin、数字、标点均按图中**视觉阅读顺序**写入；
3. 独立视觉行之间保留 `\n`，不用与图像无关的自动折行改写标签；
4. 图像退化只改变像素、不改变 GT；清晰／残墨配对的 GT 必须**逐字节相同**；
5. A/B 页面分别以实际渲染的 A 或 B 为标签，并记录正确字、rival、行号、字符位置、字体和图像条件；
6. 出现字体缺字、字体回退、字符裁断、空标签、乱码或标签与坐标重算不一致时，样本**直接拒绝**，不进入正式构建。

标签生成阶段检查字符白名单、编码往返、长度、换行和禁止字符。

## 4. 内容、字体与版式

### 4.1 内容构成

文本以规范彝文为主体，按训练目标加入受控比例的混排内容：

| 内容 | 作用 |
|---|---|
| 规范彝文全集（1,165 音节） | 主学习对象，1,165 个字符全部覆盖学习，含形近字 |
| 常用汉字 | 教彝汉混排与阅读顺序 |
| Latin 字母与常见词形 | 保持混合文档中的字母、姓名、年份、缩写 |
| 数字、编号、`①–⑳` | 覆盖页码、条目、脚注、序号格式 |
| 标点与换行 | 保持句法边界、段落结构与输出格式 |

合成文本按模板生成单行、文本区域和整页内容，覆盖标题／正文／短句／列表／脚注／带序号条目，以及不同长度、字距和行距下的阅读顺序，而非孤立字符的随机拼接。

### 4.2 字体与渲染

生成器使用四款锁定的规范彝文字体（覆盖黑体、宋体、手写风格），汉字与 Latin 使用固定 CJK/Latin 字体资源，避免系统字体漂移导致不可复现。字体按字符覆盖选择：依据字体 cmap 为每个彝文／汉字选有覆盖的字体，缺字直接报错，避免字体回退产生“豆腐块”或“标签正确但图像没字”。

版式按任务需求分配配额：单行样本用于字形与字符覆盖，区域／整页样本用于换行、混排与阅读顺序；脚注、多行和 Latin 注音样本设独立配额，避免少数高风险格式支配训练分布。

## 5. 视觉退化策略

退化用于模拟清晰打印、轻度旧化和仍可读的扫描变化，**不制造不可读图片**。每张图使用有界的退化 profile 并记录完整操作顺序。全局护栏：

| 规则 | 设置 |
|---|---|
| 覆盖范围 | 清晰、轻旧化、重但可读 |
| heavy profile 目标占比 | 30% |
| light / heavy 核心操作预算 | 最多 1 / 2 个 |
| shear / gamma | 禁用 |
| `ink` 分支 | 75% `MinFilter(3)` / 25% `MaxFilter(3)` |
| 阅读顺序 | 由版面坐标确定 |

人工抽查确认 7 类退化 profile 偏重。最终策略**保留这些类别**（不删除、不置零），仅回调强度：与同一张清晰图做固定比例插值——其中 1 类系数 `0.25`，其余 6 类 `0.80`。这样保留退化类型和曝光次数，同时避免正文接近不可读。退化后图像须通过渲染抽检，正文不可辨认、字符裁断或标签不一致者不得进入训练包。

## 6. 两阶段中的形近字训练

**形近字不是第二阶段才加入。** 两个阶段使用同一份冻结的 2,162 对通用形近字表，但训练方式不同：第一阶段在普通 OCR 样本中做完整覆盖；第二阶段在第一阶段模型上加入 A/B 最小对照数据和 token 级局部约束，继续微调。

第二阶段从**第一阶段完成 588/588 步后的合并导出模型**继续微调（不是从裸基座重训，也不是叠加另一个 adapter），新建 rank-16 LoRA 并换用新的训练参数。两个阶段的形近字来源均由训练侧字符、字形关系和固定规则独立构建，不从评估错误反向挖掘。

### 6.1 形近字对的构建

最终 2,162 个形近字对由字形分析与规则独立得到，来自四个来源：

| 来源 | 数量 | 说明 |
|---|---:|---|
| 多字体字形相似度聚类 | 1,657 | 全部 1,165 字在多款字体下渲染，在字形特征空间取 **mutual top-k**（互为前 k 近邻）的字对，跨字体一致以排除单字体渲染巧合 |
| 声调弧语言规律 | 342 | “次高调”与对应“中平调”字形相同、仅差一个笔画（声调弧），由字符体系直接推出 |
| 单字体稳定关系 | 122 | 在特定字体下稳定成对、但未达跨字体阈值的补充关系 |
| coverage-floor 覆盖补足 | 41 | 有稳定多字体近邻却未获配对的字，用换渲染尺寸的确定性规则补一对，保证覆盖不留缺口 |

聚类用的是 **mutual top-k**，不是严格“互为第一近邻”；全过程只看字形像素，不读取模型输出或评估错误。例：聚类 ꁛ↔ꄄ（0.986）、ꃽ↔ꅫ（0.983）；声调 ꀊ↔ꀉ、ꃅ↔ꃄ。多字体聚类的特征与阈值见 [`training_data/build_shape_pairs.py`](../training_data/build_shape_pairs.py)；把 2,121 扩充到最终 2,162（含单字体关系与 coverage-floor）的定稿逻辑见 [`training_data/build_shape_pairs_v3.py`](../training_data/build_shape_pairs_v3.py)。形近字的发现依据是字形与彝文字符体系，**不是 tokenizer 的 token 相似度**。

### 6.2 第一阶段：普通 OCR 上下文中的完整覆盖

第一阶段已对全部 2,162 个形近字对做强制覆盖：每对使用 3 款字体和 2 个上下文变体，共形成 `2,162 × 3 × 2 = 12,972` 个形近字覆盖事件。每个事件把一对形近字符作为相邻字符注入单行、区域或页面中的普通 OCR 文本，再用整段／整页 GT 的完整 OCR 交叉熵训练。

这一阶段的目标是让模型在不同字体、上下文和版式中都见过两个字形，建立通用辨识基础。它不是严格配对的 A/B 页面，也没有第二阶段的 byte-token 辅助损失。

### 6.3 第二阶段：受控最小对照的 A/B 版面

合成策略是**受控最小对照**：把形近字放进真实上下文，但页面的 A、B 两个版本之间只有目标字符不同、其余像素完全相同。

每个 A/B 页面组含 **8 行 = 4 款字体 × 2 种图像条件（清晰 / 轻旧化）**。每行只承载**一个**目标位置——一个形近字对，左右各至少 4 个中性字符，且 A、B 本身不出现在中性上下文中，避免上下文提前泄露答案。同一页的 8 行不重复使用形近字对；不同字对通过固定步长分散到这些行位置上。页面渲染为 A、B 两版，仅在各目标邻域把字符在 A ↔ B 间互换。

2,162 组页面与 2,162 个形近字对只是数量相同，**不是一组页面只训练一个字对**。在完整构建中，每个形近字对跨页面恰好覆盖 4 款字体 × 2 种图像条件，共 8 个匹配位置。

```text
每行固定：背景、版式、该行上下文、字体、图像条件、所有非目标像素
每行变量：该行唯一目标位置 —— A 版放 A，B 版放 B
```

这样模型只能依据**目标字符本身的细微字形差异**来区分，无法借背景、上下文或版式走捷径。规模如下：

| 检查项 | 结果 |
|---|---:|
| 形近字对 | 2,162 |
| A/B 页面组 / 图片 | 2,162 / 4,324 |
| 每页目标行数（4 字体 × 2 条件） | 8 |
| 匹配目标位置（2,162 × 8） | 17,296 |
| 有方向事件（× 2） | 34,592 |
| A、B 各作为正确字的事件数 | 17,296 / 17,296 |

每个字对在每款字体、每种图像条件下恰好出现一次（即每对出现 8 次）。**方向对称**是关键：A 与 B 各作为正确字的事件严格相等（各 17,296）。它直接针对第一阶段观察到的失败模式——模型“倾向于多加一笔或一点”（如 ꀉ ↔ ꀊ）：只朝一个方向喂样本会强化偏置，双向对称曝光才能两个方向都不混。

> 该失败模式来自第一阶段后在**合成开发集与 48 页人工诊断子集**上的错误检查，仅用于识别“形近字”这一**问题类别**；形近字对、字体、配额和训练参数仍由上述独立规则生成，不按评估错例逐个挑选字对。

### 6.4 第二阶段：byte-token 级辅助约束

本次锁定的 PaddleOCR-VL-1.6 tokenizer 对 1,165 个规范彝文字符均采用 byte fallback，每个字符固定拆成 3 个 byte token。在 2,162 个形近字对中，没有字对在第 1 个 token 分叉；1,637 对（75.7%）在第 2 个 token 分叉，525 对（24.3%）到第 3 个 token 才分叉。第二阶段利用这一结构组织辅助监督，但 token 结构不参与形近字对的发现。

第二阶段的总损失为：

```text
L = L_CE
  + 0.001  * L_byte_path
  + 0.0005 * L_full_byte_pairwise_proxy
  + 0.1    * L_non_target_consistency
```

- `L_CE` 是主损失，但对两类数据采用不同的监督范围：再次加入的第一阶段样本继续计算完整 OCR 序列交叉熵，维持混排文字、标点、换行和输出长度；新增 A/B 页面只对 8 个目标字符的 `8 × 3 = 24` 个 byte token 计算 target-only CE，把主要更新集中在形近字位置。
- `L_byte_path` 只在声明的目标字符位置生效，在正确字符 byte 路径实际发生候选分叉的位置，让正确分支与形近字表中仍未被前序 token 排除的一跳邻居进行局部竞争，而不只比较页面中声明的单个 rival；先在同一原始事件内平均，再在事件间平均，避免 token 分叉较多的字符被重复加权。
- `L_full_byte_pairwise_proxy` 比较正确字符与声明 rival 所有实际不同的 byte 位置，margin 为 `0`，并在正确字符的 teacher-forced 前缀下逐 byte 评分。它只是辅助代理项，不等同于推理阶段对完整字符联合概率进行排序。
- `L_non_target_consistency` 约束 A/B 配对样本非目标位置的中心化输出分布，并排除目标字符对应的 byte 位置，减少替换目标字符时页面其他输出发生漂移。它不使用第一阶段模型作为教师，因此不是知识蒸馏。

这些 token 级约束只在第二阶段使用，而且没有完成逐项开关消融；最终改善不能单独归因于其中任何一项。

## 7. 两阶段规模与高风险通道控制

| 阶段 | Train | Dev | 构成 |
|---|---:|---:|---|
| 第一阶段：通用混合文字适配 | 18,800 | 960 | 18,000 核心 + 400 Latin 保持 + 400 残墨成对 |
| 第二阶段：形近字专项继续微调 | 23,124 | 960 | 第一阶段 18,800 条训练样本 + 新增 4,324 条形近字 A/B |

这里的“继续微调”指模型血缘连续，**不代表两阶段沿用同一套训练参数**：

| 训练设置 | 第一阶段 | 第二阶段 |
|---|---:|---:|
| 训练起点 | 固定 PaddleOCR-VL-1.6 基座 | 第一阶段 LoRA 合并导出模型 |
| LoRA | rank 16 | 新建 rank 16，冻结语言模型主体 |
| Epoch | 2 | 1 |
| Optimizer steps | 588 | 362 |
| 学习率 / 最低学习率 | `5e-4` / `5e-5` | `1e-4` / `1e-5` |
| Warmup ratio | `0.01` | `0.03` |

第一阶段开发集为 800 核心 + 80 Latin + 80 残墨成对。第二阶段训练集再次加入第一阶段的 18,800 条训练样本，技术上属于**旧数据回放（rehearsal / replay）**；它继续约束普通彝文、汉字、Latin、数字、标点、换行和整页输出，防止新增专项数据把模型推成局部字符分类器。

因此第二阶段同时包含两层连续关系：**模型层面**继承第一阶段合并权重并换用新 LoRA、新参数继续微调；**数据层面**回放第一阶段训练样本，再加入新的 A/B 形近字样本。两者并不矛盾。现有训练没有完成“只用 4,324 条 A/B”与“18,800 条旧样本 + 4,324 条 A/B”的独立对照，因此不能宣称旧数据占比或两类数据比例已经最优。

开发集保持第一阶段冻结的 960 条，不写入 A/B 训练页面或真实评估集，且不参与训练期 checkpoint 选择或提前停止。

高风险通道按独立配额进入训练，不因“看起来更真实”而无限扩张：

| 通道 | 构建规则 |
|---|---|
| Latin / 注音 | 保留混排与必要字典格式，限制比例，防止默认追加 Latin 尾巴 |
| 脚注与符号 | 少量覆盖序号、脚注、符号布局，防止公式化片段成为默认输出 |
| 多行区域 | 保留完整换行训练但设上限，防止单行被强制分段 |
| 残墨成对 | 清晰／退化文本逐字节一致，退化只改图像外观 |
| 形近字 A/B | 同一事件内部对称采样，不把某方向过采样成先验 |

超上限的样本不通过合并门，而不是事后靠损失权重补救。

## 8. 数据格式、质量控制与污染审计

训练样本为 JSONL `messages/images` 格式，标签作为 assistant 内容：

```json
{
  "messages": [
    {"role": "user", "content": "<image>OCR:"},
    {"role": "assistant", "content": "<generated transcription>"}
  ],
  "images": ["<relative image path>"]
}
```

正式合并前至少执行：图像存在且可解码、路径在构建目录内；标签非空、Unicode 往返稳定、无 `�` 或公式化片段；图像哈希／sample ID／完整目标文本在 train/dev 间不重复；A/B 记录成对且方向齐全；每类样本、字体、规则、视觉变化的来源字段可追溯；运行污染审计确认输入枚举、路径根、来源 token 和 manifest 中无评估图片、GT、预测或错误统计。

## 9. 构建方法的核心机制

这套构建器把数据分布、视觉渲染和审计约束放进同一条可复现流水线：

| 机制 | 程序实现 | 解决的问题 |
|---|---|---|
| 每样本确定性随机 | 从总 seed 派生文本／版式／渲染／退化子序列 | 渲染顺序变化不改变样本；重复构建可逐样本复现 |
| 覆盖感知字体选择 | 按字体 cmap 选有覆盖的锁定字体，缺字报错 | 避免字体回退、豆腐块、“标签对但图像没字” |
| 先规划后采样再渲染 | 先定类别／粒度／版式／退化配额，再生成文本与图片 | 防止随机采样把训练集挤成单一字体／版式／退化 |
| 坐标化阅读顺序 | 先生成带坐标的 block/line，再按阅读顺序写标签与换行 | 让整页 OCR 学到真实顺序而非孤立行拼接 |
| 训练／开发几何隔离 | train 与 dev 使用不同几何 profile 并记录 split | 降低开发集只记住训练版式的风险 |
| 退化策略锁定 | profile、操作顺序、`ink_mode`、强度由 schema/SHA 校验的策略文件约束 | 防止随手改参、删过重类别或恢复无限叠加退化 |
| 哈希去重 + fail-closed | 每次渲染检查图像哈希，碰撞／缺图／坏图直接阻止发布 | 避免“行数够但图片重复”与静默坏图 |
| provenance + 审计门 | 清单记录代码、字体、字符表、规则、行数、SHA-256，正式训练前跑污染审计 | 让每个训练包能回答“由什么生成、是否混入评估、是否可复现” |

## 10. 核心判断

这套策略不是“尽可能多生成图片”，而是把覆盖需求拆成可审计的信号：

1. 用规范字符和锁定字体，让 1,165 个规范彝文字符全部被学到；
2. 用混合文字和完整版式教模型按阅读顺序输出；
3. 用受控退化覆盖真实扫描变化，但拒绝不可读尾部；
4. 第二阶段在第一阶段合并模型上，用对称 A/B 形近字事件补细粒度判别，同时保留第一阶段训练样本对完整 OCR 能力的约束；
5. 用独立 build、哈希、配额与污染审计保证训练数据可复现、可解释且不侵入评估集。

因此，合成数据的价值是**补足真实材料无法稳定覆盖的分布**，而不是替代真实评估，也不是用合成样本数量直接证明模型能力。真实能力由独立的 1,030 张评估集衡量，其结果见 [评估结果分析](EVALUATION_RESULT_ANALYSIS.md) 与 [模型与训练](MODEL_AND_TRAINING.md)。
