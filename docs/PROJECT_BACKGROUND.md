# 项目背景与任务定义

本文档对应主提交稿中的“项目概览、场景稀缺性与任务复杂度”。它回答三个问题：为什么选择规范彝文 OCR、这个任务难在哪里、模型最终要解决什么实际问题。

## 项目概览

`规范彝文 OCR / NuosuBburma OCR` 基于 `PaddleOCR-VL-1.6 (0.9B)` 进行 LoRA 微调，面向规范彝文（ꆈꌠꁱꂷ / Nuosu Bburma）真实场景 OCR。

项目目标是把书籍扫描页、裁切行图、文字区域、手写稿和彝汉混排材料中的规范彝文转成可复制、可检索、可继续加工的 Unicode 文本。它既服务于文献数字化，也服务于学习、检字、注音、校对和后续 NLP 语料建设。

这个项目从真实旧书出发。先让模型学会规范彝文字形，再补切图、识别、混排、输出稳定性和评估复查这些实际环节。

## 场景稀缺性

规范彝文诞生于 1980 年，已经形成稳定的标准字符体系，承载了教材、诗歌、民族研究文献和日常教学资料。但在数字化层面，大量资料仍停留在图片、扫描件和纸质书稿中，无法稳定搜索、复制和整理。

这会带来几个直接问题：

| 问题 | 影响 |
|---|---|
| 学习依赖检字 | 学习者需要反复查字形、读音和含义，学习成本高 |
| 资料难以复制 | 教材、旧书和文献不能方便整理、引用和再加工 |
| 检索能力缺失 | 民族文献和教学资料难以进入数字检索系统 |
| 后续 NLP 缺语料 | 转写、注音、翻译和语言模型训练缺少高质量文本入口 |
| 通用 OCR 基本不可用 | 现有通用 OCR 很难直接识别规范彝文，往往需要重新建立数据和评估入口 |

这也是项目最初的起点：可用工具很少，但真实资料数字化、教学检字和语料建设的需求一直在。

## 任务定义

```text
输入：包含规范彝文或规范彝文混排内容的图像
输出：图像中可见文本的 Unicode 转写，尽量保留混排关系和基本标点
```

真实使用路径如下：

```text
用户上传 PDF 或图片
-> 页面渲染或读取
-> 自动切图或人工裁剪文字区域
-> 对 page / region / line 执行 OCR
-> 导出文本、CSV 或 JSON
-> 需要时继续做注音和人工校对
```

本模型支持整页图像、文本区域和裁切行图输入。当前最稳定的使用方式通常是 `line / region OCR`；复杂整页文档在版面较密、手写、多栏、脚注、注音块或图文混排较强时，建议配合版面分析、切图流程或人工复核。

## 任务复杂度

规范彝文 OCR 的复杂度来自真实资料本身。单行识别准确，不等于真实资料整理可用。

| 复杂度 | 具体表现 | 对模型的影响 |
|---|---|---|
| 字符复杂度 | 规范彝文覆盖 1165 个字符，形近字多，差异常集中在细小笔画、方向、开口和局部结构 | 低分辨率、旧印刷和压缩会放大形近字混淆 |
| 视觉域复杂度 | 新印刷 PDF、旧书扫描、页面照片、手机照片、手写材料噪声不同 | 模型不能只适应单一字体或单一本书 |
| 结构复杂度 | 真实资料包含整页、区域、单行、短诗式排布、脚注块、注音块和段落图 | 多行区域容易漏行，复杂整页需要切图或版面分析配合 |
| 混排复杂度 | 彝文、汉语、数字、脚注符号、拉丁注音和标点会同时出现 | 容易出现 LaTeX 化、ASCII 尾巴、长输出等输出空间漂移 |

评估和选模时，这些问题都会单独看。清晰单行只是其中一种输入，不能代表真实资料整理。

## 当前能力边界

- 支持 page / region / line 图像输入。
- 清晰印刷行图和常见彝汉混排是当前最稳定场景。
- 复杂整页、密集多行 region、脚注/注音块建议配合切图流程或人工复核。
- 手写已有一定泛化能力，但仍明显弱于印刷体。
- 尚未进行专门的端侧/移动端优化。
