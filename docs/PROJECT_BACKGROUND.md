# 项目背景与任务定义

本文档说明为什么选择规范彝文识别、任务难点在哪里，以及模型面向哪些实际使用场景。

## 项目概览

`规范彝文识别 / NuosuBburma OCR` 基于 `PaddleOCR-VL-1.6 (0.9B)` 进行 LoRA 微调，是一个面向规范彝文（ꆈꌠꁱꂷ / Nuosu Bburma）真实场景的 PaddleOCR-VL 衍生模型。

项目目标是把书籍扫描页、裁切行图、文字区域和手写拍照稿中的规范彝文转成可复制、可检索、可校对的 Unicode 文本。后续可用于检字、注音、检索和语料整理。

项目从真实旧书和资料图片出发。模型先学习规范彝文字形，再处理页面切割、混排识别、输出风险和评估复查。

## 场景稀缺性

规范彝文诞生于 1980 年，已经形成稳定的标准字符体系，承载了教材、诗歌、民族研究文献和日常资料。但在数字化层面，大量资料仍停留在图片、扫描件和纸质书稿中，无法稳定搜索、复制和整理。

这类场景的稀缺性不只来自“数据少”，还来自工具不足。通用文字识别工具通常不认识规范彝文，公开评估集也很少。研究者和资料整理者即使有资料，也常常只能人工录入。

本项目把问题限定在可检查的文字识别工作流：真实资料输入、人工核对评估集、模型识别、页面文本合并和后续校对。

这会带来几个直接问题：

| 问题 | 影响 |
|---|---|
| 检字依赖人工 | 字形、读音和含义需要反复人工核对，整理成本高 |
| 资料难以复制 | 教材、旧书和文献不能方便整理、引用和再加工 |
| 检索能力缺失 | 民族文献和资料图片难以进入数字检索系统 |
| 后续 NLP 缺语料 | 转写、注音、翻译和语言模型训练缺少高质量文本入口 |
| 通用文字识别基本不可用 | 现有通用文字识别工具很难直接识别规范彝文，往往需要重新建立数据和评估入口 |

项目起点很简单：可用工具少，但资料数字化、检字、注音和语料整理都有需求。文字识别先把图片资料转成文本，后续才能继续校对、注音、检索和训练。

## 任务定义

```text
输入：包含规范彝文或规范彝文混排内容的图像
输出：图像中可见文本的 Unicode 转写，尽量保留混排关系和基本标点
```

真实使用路径如下：

```text
用户上传 PDF 或图片
-> 页面渲染或读取
-> 自动页面切割或人工裁剪文字区域
-> 对单行图或区域图执行识别，必要时单独观察整页识别
-> 导出文本、CSV 或 JSON
-> 需要时继续做注音和人工校对
```

本模型可以输入整页图像、文本区域和裁切行图。当前最稳定的使用方式通常是单行图和区域图识别。

直接整页识别用于观察整页边界、阅读顺序和换行风险，不作为复杂整页稳定交付的主要路径。复杂整页文档建议配合页面切割流程或人工复核。

这个定义刻意保守：先把单行图和区域图能力做稳，再用页面切割把整页问题拆开。对低资源文字来说，可复查比一次性包装成完整产品更重要。

## 任务复杂度

规范彝文识别的复杂度来自真实资料本身。单行识别准确，不等于真实资料整理可用。

| 复杂度 | 具体表现 | 对模型的影响 |
|---|---|---|
| 字符复杂度 | 规范彝文覆盖 1165 个字符，字符差异常集中在细小笔画、方向、开口和局部结构 | 低分辨率、旧印刷和压缩会放大字符边界不稳定 |
| 视觉域复杂度 | 新印刷 PDF、旧书扫描、页面照片、手机照片、手写拍照材料噪声不同 | 模型不能只适应单一字体或单一本书 |
| 结构复杂度 | 真实资料包含整页、区域、单行、短诗式排布、脚注块、注音块和段落图 | 多行区域容易漏行，复杂整页需要页面切割配合 |
| 混排复杂度 | 彝文、汉语、数字、脚注符号、拉丁注音和标点会同时出现 | 容易出现公式化片段、多余 ASCII 或异常长输出 |

评估和选模时，这些问题都会单独看。清晰单行只是其中一种输入，不能代表真实资料整理。项目保留单行图、区域图、整页图、手写、旧印刷和混排分组，就是为了让每一种能力各自暴露，而不是被一个平均分掩盖。

## 当前能力边界

- 可以输入整页图、区域图和单行图。
- 清晰印刷行图和常见彝汉混排是当前最稳定场景。
- 复杂整页直接识别容易出现非标注换行、阅读顺序偏差或段落边界误判，建议配合页面切割流程或人工复核。
- 手写拍照资料已纳入独立观察，结果与印刷体分开解读。
- 尚未进行专门的端侧/移动端优化。
