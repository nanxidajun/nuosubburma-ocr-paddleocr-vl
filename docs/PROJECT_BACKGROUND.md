# 项目背景与任务定义

本文档说明为什么做规范彝文 OCR、希望模型做到什么、任务难点在哪里，以及模型面向哪些真实场景。

## 为什么做规范彝文 OCR

规范彝文（ꆈꌠꁱꂷ / Nuosu Bburma）是一门**仍在使用、仍在创作的活语言文字**，北部彝语（诺苏语）使用者约 300 万。它自 1980 年形成稳定的标准字符体系，承载教材、诗歌、民族研究文献和日常资料——但在数字化层面，大量内容仍停留在图片、扫描件和纸质书稿里，无法稳定搜索、复制和整理。

需求是真实且持续的：

- **旧书资料**——大量内容仍停留在不可检索的图片里；
- **教学考试**——教材、课堂与考试需要可复制、可排版的文本；
- **新的创作**——母语创作持续产生，需要顺畅进入数字流程；
- **数字缺口**——主流 OCR 模型尚不能稳定识别规范彝文。

这类场景的稀缺性不只来自“数据少”，更来自“工具缺”。通用文字识别工具通常不认识规范彝文，公开评估集也很少，研究者和资料整理者即使有资料，也常常只能人工录入。

| 问题 | 影响 |
|---|---|
| 检字依赖人工 | 字形、读音和含义需反复人工核对，整理成本高 |
| 资料难以复制 | 教材、旧书和文献不能方便整理、引用和再加工 |
| 检索能力缺失 | 民族文献和资料图片难以进入数字检索系统 |
| NLP 缺语料 | 转写、翻译和语言模型训练缺少高质量文本入口 |
| 通用 OCR 基本不可用 | 现有工具很难直接识别规范彝文，往往需从零建立数据与评估入口 |

OCR 是这条链路的第一步：先把图片资料转成可核验的 Unicode 文本，才能继续校对、检索、语料建设和低资源模型训练。

## 我们希望模型做到什么

围绕真实使用，模型的目标定为三条：

1. **整页输入与输出。** 整页图片直接进入，按视觉阅读顺序输出整页文字，保留必要的行结构——不切行、不切块、不做多模型路由。
2. **混排识别。** 彝文、汉字、拉丁字母、数字和标点在同一次识别中一起输出。
3. **满足多种场景。** 需求不止清晰文档，还有旧书扫描、屏幕拍照、手写和更多现实材料。

这三条决定了模型是一个**端到端单模型**：一张图进、一段完整 Unicode 文本出。

## 任务定义

```text
输入：包含规范彝文或规范彝文混排内容的图像
输出：图像中可见文本的 Unicode 转写，按阅读顺序保留混排关系、行结构和基本标点
```

真实使用路径：

```text
用户上传 PDF 或图片
  → 页面渲染或读取
  → 整幅图片（行图、区域图或整页）直接输入单一模型
  → 端到端输出该图的完整 Unicode 文本
  → 生成文本、CSV 或 JSON
  → 人工校对
```

行图、文本区域和整页都直接输入，由单一模型一次读出，不做页面切割或多模型路由，最终评估也在整页图片上直接进行。书籍扫描与清晰印刷最稳定；复杂整页、照片、手写和低清材料误差更高，建议人工复核。

这个定位刻意保守：以可核验的整体识别结果为准，而不是把不稳定的场景包装成完整产品。对低资源文字来说，可核验比一次性交付更重要。

## 任务复杂度

规范彝文识别的复杂度来自真实资料本身。单行识别准确，不等于真实资料整理可用。

| 复杂度 | 具体表现 | 对模型的影响 |
|---|---|---|
| 字符复杂度 | 1,165 个规范彝文音节，字符差异常集中在细小笔画、方向、开口和局部结构 | 低分辨率、旧印刷和压缩会放大字符边界不稳定 |
| 视觉域复杂度 | 新印刷 PDF、旧书扫描、页面照片、手机照片、手写拍照的噪声各不相同 | 模型不能只适应单一字体或单一本书 |
| 结构复杂度 | 真实资料包含整页、区域、单行、短诗式排布、脚注块、注音块和段落图 | 多行区域容易漏行，复杂整页的阅读顺序和段落边界更难稳定 |
| 混排复杂度 | 彝文、汉字、数字、脚注符号、拉丁字母和标点会同时出现 | 容易出现公式化片段、多余 ASCII 或异常长输出 |

评估和选模时，这些问题都会分组单独看。项目保留书籍、手写、屏幕、实景等采集场景，以及纯彝文、彝汉、彝汉拉丁等文字构成分组，就是为了让每一种能力各自暴露，而不是被一个平均分掩盖。

## 当前能力边界

- 整幅图片端到端识别，无需切行或切块；书籍扫描与清晰印刷最稳定。
- 复杂整页、屏幕照片、实景照片和长手写段落误差更高，建议人工复核。
- 个别高频单向形近字混淆仍然存在。
- 手写拍照资料已纳入独立观察，结果与印刷体分开解读。
- 尚未进行专门的端侧 / 移动端优化。
